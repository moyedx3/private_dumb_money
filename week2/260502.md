# Zcash 프로젝트 논의 회의록

## 1. 회의 목적

오늘 회의는 우리 팀이 앞으로 진행할 **Zcash 기반 프로젝트의 방향성**을 잡기 위해 진행했다. 꼭 해커톤 제출을 전제로 한 것은 아니며, 기존 Zcash 관련 프로젝트들을 살펴보고 우리가 만들 수 있는 제품적 기회와 차별화 지점을 논의했다.

이 문서는 회의에 참석하지 못한 팀원에게 논의 흐름을 공유하고, 이후 아이디어를 다시 검토하기 위한 복기용 회의록이다.

---

## 2. 핵심 요약

기존 Zcash 프로젝트들을 살펴본 결과, 많은 프로젝트가 비슷한 방향으로 수렴하고 있었다. 대표적으로는 송금 UX 개선, 지갑 사용성 개선, x402 결제, private paywall, merchant payment, private receipt, selective disclosure 등이 있었다.

회의에서 가장 중요하게 본 질문은 다음이었다.

> **왜 이 프로젝트는 Zcash를 써야 하는가?**

단순히 "Zcash로 결제한다"거나 "private payment를 제공한다"는 설명만으로는 충분하지 않다고 보았다. 다른 체인이나 일반 결제수단으로도 가능한 아이디어라면 Zcash 프로젝트로서의 설득력이 약하기 때문이다.

따라서 방향성은 다음처럼 정리되었다.

- 완전히 새로운 암호학 프로토콜을 만드는 쪽으로 바로 가기보다는, 앱 레벨에서 시작한다.
- 다만 앱의 핵심 이유가 Zcash의 고유 구조와 연결되어야 한다.
- 특히 note, shielded transaction, selective disclosure, viewing key 같은 개념이 제품의 차별점으로 이어져야 한다.
- 결제뿐 아니라 **Zcash memo를 키교환 채널로 활용하는 검열저항적 P2P 메시징 프로토콜** 가능성도 함께 검토했다.
- "프라이버시 결제" 자체보다, 프라이버시 때문에 생기는 실제 사용 문제를 푸는 쪽이 더 가능성이 있어 보인다.

---

## 3. 기존 프로젝트를 보며 느낀 점

기존 프로젝트들은 대체로 다음 범주에 많이 걸쳐 있었다.

- Zcash 송금 UX 개선
- 지갑 사용성 개선
- Near Intents 등을 활용한 송금
- x402와 Zcash 결합
- private paywall
- shielded merchant payment
- private receipt
- selective disclosure
- note 기반 claim 또는 airdrop류 아이디어
- private messaging 또는 wallet-to-wallet communication
- Zcash memo를 활용한 key exchange / message bootstrap

이 목록을 보면서 나온 공통된 판단은, **단순한 결제 또는 송금 개선만으로는 차별화가 어렵다**는 것이었다. 특히 "Zcash를 더 쉽게 쓰게 하자" 또는 "결제를 비공개로 만들자"는 방향은 이미 많이 시도되고 있거나 자연스럽게 떠올릴 수 있는 아이디어에 가깝다.

우리 팀이 프로젝트를 설계한다면, 기존 범주와 겹치더라도 더 구체적인 사용 맥락과 명확한 문제 정의가 필요하다.

---

## 4. 논의의 기준

아이디어를 평가할 때 다음 기준을 중요하게 보기로 했다.

- Zcash를 쓰는 필연성이 있는가?
- 프라이버시가 부가 기능이 아니라 제품의 핵심인가?
- 투명한 EVM 체인이나 일반 결제 시스템에서는 구현하기 어려운가?
- Zcash의 shielded note, viewing key, selective disclosure 같은 구조가 실제 사용자 경험으로 이어지는가?
- 사용자가 굳이 ZEC 또는 Zcash 생태계를 써야 하는 이유가 있는가?
- 기존 private payment, private paywall, receipt 프로젝트와 무엇이 다른가?

특히 "왜 Zcash여야 하는가?"에 답하지 못하는 아이디어는 우선순위를 낮게 보기로 했다.

---

## 5. 주요 논의 내용

### 5.1 note를 제품적으로 활용할 수 있는가

Zcash의 고유한 구조인 **note**를 단순한 잔액 단위가 아니라 제품 기능으로 해석할 수 있는지 논의했다.

회의 중 나온 관점은 다음과 같다.

> note를 "돈"이 아니라 "비공개 권리 증명"처럼 사용할 수 있지 않을까?

예를 들어 특정 note를 가진 사람만 어떤 권리를 행사할 수 있게 하되, 그 사람이 누구인지나 어떤 note를 가지고 있는지는 드러내지 않는 방식이다.

가능한 예시는 다음과 같았다.

- private coupon
- private event ticket
- private allowlist
- private membership
- private receipt
- note-based private claim
- selective disclosure 기반 결제 증명

다만 실제 앱 레벨에서 note에 얼마나 접근할 수 있는지, SDK나 지갑이 어떤 기능을 지원하는지는 아직 확인이 필요하다.

### 5.2 x402와 Zcash의 결합

x402와 Zcash를 결합하는 프로젝트는 이미 존재하거나 논의된 것으로 보였다. 따라서 단순히 "API 요청 시 ZEC로 결제한다"는 구조만으로는 차별성이 부족하다.

더 설득력 있는 방향은 x402를 결제 요청 또는 트리거로 보고, Zcash를 private settlement 또는 private access proof 레이어로 쓰는 것이다.

예를 들면 다음과 같은 흐름이다.

1. 사용자가 API나 콘텐츠 접근을 요청한다.
2. 서버가 결제를 요구한다.
3. 사용자가 shielded ZEC로 결제한다.
4. 이후 사용자는 wallet identity를 직접 드러내지 않고 접근권을 증명한다.

핵심은 "결제를 했다"는 사실은 증명하되, 지갑을 곧바로 사용자의 identity로 만들지 않는 것이다.

### 5.3 Zcash의 강점보다 약점을 제품 문제로 보기

Zcash의 프라이버시는 강점이지만, 실제 제품 관점에서는 여러 문제가 생긴다.

예를 들어:

- 결제 여부 확인이 어렵다.
- 환불이나 CS 처리가 어렵다.
- 영수증이나 증빙을 어떻게 제공할지 애매하다.
- merchant accounting이 어렵다.
- 감사, 정산, 규제 대응과 private payment가 충돌할 수 있다.
- 사용자가 selective disclosure를 이해하고 쓰기 어렵다.

이 문제를 단순히 단점으로만 보지 않고, 오히려 제품 기회로 볼 수 있다는 이야기가 나왔다. 즉, "더 프라이빗한 결제"가 아니라 **private but accountable payment**를 가능하게 하는 도구가 더 현실적인 방향일 수 있다.

### 5.4 Zcash memo 기반 메시징 프로토콜 가능성

결제 외에도 **메시징 프로토콜**을 프로젝트 방향으로 볼 수 있는지 논의했다. 이 논의는 단순 채팅 앱을 만들자는 이야기라기보다, 기존 메신저의 검열 가능성과 키교환 문제를 Zcash의 memo 기능으로 우회할 수 있는지에 가까웠다.

논의된 문제의식은 다음과 같다.

- Telegram식 P2P 메시징도 결국 중앙화된 서비스 계층에 의존하기 때문에 검열 가능성이 남는다.
- P2P 메신저에서 가장 까다로운 부분 중 하나는 처음 상대를 찾고 키를 교환하는 과정이다.
- 이 키교환 또는 bootstrap 정보를 Zcash shielded transaction의 memo에 담으면, 중앙화 메신저 서버를 거치지 않고도 통신을 시작할 수 있지 않을까?
- Zcash memo는 결제와 함께 전달되는 private data channel처럼 볼 수 있으므로, 메시징 프로토콜의 시작점으로 활용할 여지가 있다.

아이디어의 기본 흐름은 다음과 같다.

1. 사용자가 Zcash shielded transaction을 보낸다.
2. transaction memo 안에 P2P 통신을 시작하기 위한 public key, rendezvous 정보, 또는 암호화된 초대 정보를 담는다.
3. 상대방은 memo를 읽고 별도의 중앙화 메신저 없이 세션을 시작한다.
4. 이후 실제 메시지는 P2P 네트워크 또는 별도 transport 위에서 end-to-end encrypted 방식으로 주고받는다.

이 방향은 "Zcash로 메시지를 보낸다"가 아니라, **Zcash memo를 검열저항적인 키교환 및 연결 수립 레이어로 사용한다**는 점이 핵심이다. Telegram 같은 기존 메신저가 P2P 기능을 제공하더라도 계정, discovery, relay, abuse control 등 서비스 계층에서 차단될 수 있는데, Zcash memo를 이용하면 최소한 최초 연결 정보의 전달을 Zcash 네트워크 위에 올릴 수 있다.

따라서 회의에서는 이 아이디어가 꽤 괜찮은 방향일 수 있다고 보았다. 다만 실제 검열저항성이 얼마나 강한지는 transport, metadata leakage, spam 방지, memo 크기 제한, 지갑 UX에 따라 달라지므로 추가 검증이 필요하다.

### 5.5 개발 층위

프로젝트를 어느 깊이에서 만들지에 대해서도 논의했다.

완전히 프로토콜 레벨로 들어가면 Zcash를 쓰는 이유는 강해지지만, 구현 난이도가 크게 올라간다. 반대로 앱 레벨에만 머물면 빠르게 만들 수 있지만, 기존 프로젝트와 차별화가 약해질 수 있다.

현재는 다음과 같은 중간 지점이 적절해 보인다는 의견이 있었다.

> 앱 레벨에서 만들되, Zcash의 프로토콜적 특징이 제품의 핵심 이유가 되도록 설계한다.

즉, 단순 지갑 UX나 결제 UX가 아니라 note, shielded transaction, selective disclosure, viewing key 등이 실제 제품 흐름 안에서 의미를 가져야 한다.

## 6. 가능성 있어 보인 방향

회의에서 명확히 하나의 아이디어를 결정한 것은 아니다. 다만 다음 방향들이 비교적 더 가능성이 있어 보였다.

### 6.1 Private Access Pass

Zcash shielded payment로 접근권을 사고, 이후에는 사용자가 wallet identity를 드러내지 않고 API, 콘텐츠, 서비스에 접근하는 구조다.

핵심 메시지는 다음과 같다.

> 결제는 증명하지만, 지갑을 identity로 만들지는 않는다.

일반 crypto paywall은 결제 증명과 wallet identity가 강하게 연결될 수 있다. 이 방향은 결제 여부는 확인하되, 이후 접근권 행사는 더 분리된 방식으로 만드는 데 초점이 있다.

### 6.2 Private but Accountable Payment Toolkit

사용자는 프라이버시를 유지하고, merchant나 서비스 제공자는 필요한 수준의 영수증, 정산, 감사 자료를 확보할 수 있게 하는 도구다.

이 방향은 Zcash의 프라이버시 자체보다, 프라이버시 때문에 생기는 실무 문제를 해결하는 데 초점이 있다. merchant payment, receipt, selective disclosure와 관련 있지만, 단순 영수증 앱이 아니라 "프라이버시와 정산 가능성 사이의 균형"을 제품 문제로 삼는다는 점이 중요하다.

### 6.3 구체적 use case 기반 private payment

기술 자체가 새롭지 않더라도, 사용 맥락을 구체적으로 잡으면 차별화할 수 있다.

예시로는 다음이 나왔다.

- AI agent의 유료 API 사용 내역 보호
- journalist/source 간 private tip과 proof
- private research dataset access
- crypto alpha API 접근권
- DAO 내부 익명 기여자 보상
- whistleblower 자료 제출 보상
- private membership 또는 private event access

이 경우 중요한 질문은 "private payment가 가능한가?"가 아니라 **누구의 어떤 위험을 줄이는 private payment인가?**이다.

### 6.4 Zcash Memo Key Exchange Messenger

Zcash memo를 이용해 P2P 메시징의 키교환 또는 초대 정보를 전달하는 프로토콜이다.

기존 메신저는 P2P 메시징을 제공하더라도 계정, 서버, discovery, relay 같은 지점에서 검열될 수 있다. 이 아이디어는 최초 키교환 정보를 Zcash shielded memo에 넣어 전달함으로써, 중앙화된 메신저 인프라 없이도 상대와 통신 세션을 시작하는 것을 목표로 한다.

핵심은 "또 하나의 익명 채팅 앱"이 아니라 **Zcash memo를 censorship-resistant key exchange layer로 쓰는 것**이다. 실제 메시지 전송은 별도 P2P transport에서 처리하고, Zcash는 상대방 발견과 세션 시작을 위한 최소 정보를 안전하게 전달하는 역할을 맡는다.

회의에서는 이 방향이 Zcash를 써야 하는 이유가 비교적 분명하다고 보았다. 특히 Zcash의 memo가 단순 결제 부가 정보가 아니라, 검열저항적 communication bootstrap으로 확장될 수 있다는 점이 차별화 포인트다.

## 7. 아직 확인해야 할 점

다음 내용은 추가 조사가 필요하다.

- Zcash note를 실제 앱 레벨에서 얼마나 직접적으로 활용할 수 있는가?
- SDK에서 note ownership, memo, viewing key, selective disclosure를 어느 정도 지원하는가?
- x402와 Zcash를 이미 결합한 프로젝트들은 정확히 어디까지 구현했는가?
- Private Access Pass가 기존 private paywall과 무엇이 다른가?
- note-based private claim을 현실적인 제품 범위로 단순화할 수 있는가?
- Zcash의 프라이버시를 유지하면서 merchant accounting을 가능하게 하는 방법이 있는가?
- "프라이버시 결제"가 아니라 "wallet identity를 만들지 않는 access proof"로 설명할 수 있는가?
- Zcash memo에 키교환 또는 초대 정보를 담는 것이 실제로 가능한가?
- memo 크기 제한, 암호화 방식, 지갑 지원 범위는 어느 정도인가?
- memo 기반 키교환 이후 실제 메시지 transport는 무엇을 사용할 것인가?
- Telegram식 P2P와 비교했을 때 어떤 지점에서 검열저항성이 강해지는가?
- metadata leakage, spam, abuse, rendezvous server 의존성 문제를 어떻게 줄일 수 있는가?

## 8. 다음 논의 때 보면 좋은 것

다음 회의에서는 아이디어를 더 넓게 발산하기보다, 후보 방향별로 실제 구현 가능성과 기존 프로젝트와의 차이를 확인하는 것이 좋다.

우선순위는 다음과 같다.

1. 기존 Zcash + x402 / private paywall / merchant payment 프로젝트 조사
2. Zcash SDK와 지갑에서 앱 개발자가 접근할 수 있는 기능 확인
3. note, viewing key, selective disclosure를 실제 제품 흐름에 넣을 수 있는지 검토
4. Private Access Pass와 Private but Accountable Payment Toolkit의 차별점 정리
5. Zcash memo 기반 key exchange가 기술적으로 가능한지 검토
6. 우리 팀이 단기간에 만들 수 있는 MVP 범위 정의

## 9. 한 줄 정리

오늘 논의의 핵심은 단순히 Zcash로 결제하는 제품이 아니라, **Zcash의 프라이버시 구조 때문에 가능해지는 사용자 경험과 제품 문제를 찾아야 한다**는 것이다. 특히 Zcash memo를 P2P 메시징의 키교환 레이어로 활용하면, 기존 메신저보다 검열저항적인 communication bootstrap을 만들 수 있다는 아이디어가 유력한 방향으로 떠올랐다.
