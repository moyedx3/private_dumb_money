## 2. 리서치 공유

### 2.1 Sipher & SIP Protocol — 발화자 2

- 구조 요약: 최하단의 **SIP Protocol(SDK)** 위에 **Sipher**, 그 위에 **AI 에이전트**가 얹히는 다이어그램.
  - SIP Protocol은 기존 암호학 프리미티브들을 "짬뽕"으로 묶어 둔 SDK 성격.
  - Sipher는 AI 에이전트가 프라이빗 페이먼트를 "딸깍" 한 번에 호출할 수 있도록 노출하는 툴/스킬셋 레이어.
- GPC뿐 아니라 스텔스 기능을 가진 여러 프라이버시 프로젝트들에 "발을 걸쳐" 두는 형태. 해커톤 쌀먹(다수 대회 동시 제출) 전략 색이 강함.
- **Decshield(=Deshield)** 경로는 RPC 한 줄 위임이라 사실상 큰 의미 없음. GPC가 단지 path 중 하나로 들어가 있어 레이더에 잡힌 것.
- 테크니컬 인사이트보다는 **해커톤 전략** 측면에서 참고할 만함.
- 결론: 우리가 SDK를 직접 만들거나 클로드용 스킬셋을 만드는 방향은 **우리 방향과 맞지 않음**.

### 2.2 Pay Anyone Legend (GPC × NEAR Intents) — 발화자 1

- `pay-anyone-legend` 폴더의 팀 워크스루 MD 기반 설명.
- 동작 흐름:
  1. "Pay olipan…" 같은 자연어 입력을 AI 에이전트가 파싱 (실제로는 LLM이라기보다 DB 룩업/규칙 기반).
  2. amount / 통화 / 체인 정보를 추출해 **One-Click API**(NEAR 측)에 요청.
  3. One-Click API가 세션 deposit 주소를 발급. 사용자가 해당 주소로 송금.
  4. 별도 cron이 폴링하여 success 감지 시 NEAR MPC로 USDC TX 서명 → Base 체인으로 트랜잭션.
  5. `x402 payment header`로 매칭되면 paywall 해제.
- MPC 사용 이유: 사용자별 키를 선형 증가시키지 않고 **부분키 조합으로 매번 새 주소를 도출**하기 위한 키 관리 목적.
- 추가 확인:
  - One-Click이 솔버 역할. NEAR Intents를 쓰면 더 글로벌하게 "아무 체인이나 지불"을 만들 수 있을 것.
  - NEAR Chain Signature(MPC)는 NEAR가 직접 구현. 메타 트랜잭션 + 가스비 대납 + 타 체인 지갑 조정.
  - 즉, 한 흐름에 **솔버를 두 번** 끼는 구조 (Zcash↔NEAR, NEAR↔Base).
- 한계: AI 에이전트가 결제 의도를 파싱하는 시점에 **프라이버시 훼손**이 발생할 수 있음. 로컬 실행 여부는 불명확.

### 2.3 Aztec Bridge (Zashi 연동) — 발화자 3

- 다른 브릿지들과 달리 1:1 매핑이 아닌 **Burn & Mint 방식**.
- 흐름 (Aztec ↔ Zcash):
  1. 유저가 Bridge API의 `create endpoint` 호출 → 서버가 (수량, secret) 해시를 Aztec Bridge 컨트랙트에 등록. Ticket ID 발급.
  2. 유저는 Zcash Shielded TX의 메모에 Ticket ID를 박아 전송.
  3. **Zcash Watcher**가 폴링으로 메모를 감지 → `deposit.json`에 로그.
  4. API가 그것을 폴링 → 브릿지 컨트랙트에 `registerTicket` 호출.
  5. 유저가 `claim` 요청 → Aztec에서 검증 후 mint 발생.
  6. 출금은 동일 흐름의 역방향.
- Aztec은 EVM이 아니라 **Noir** 기반. zk 회로 안에서 secret을 숨길 수 있음.
- 검증 = 동일한 값을 재생산할 수 있는지 확인.
- 한계 / 문제점:
  - **운영자가 모든 출금 요청을 볼 수 있음.** 수량도 해시 처리되지만 운영자는 누가 뭘 했는지 알 수 있음.
  - 무조건 운영자를 신뢰해야 하는 구조 (Zcash Watcher가 사실상 출금 권한자).
  - 그래도 **메모 활용 패턴** 자체는 우리가 써먹을 만한 부분.
- 해커톤 시상: 약 $1,500 수상. (참고: SIP Protocol, Pay Anyone Legend도 별도 수상.)

### 2.4 Zchat (Zashi 기반 메신저) — 발화자 4

- 빌드는 실패해 직접 실행해 보지 못함. 코드만 분석. (안드로이드 위주, 웹은 껍데기뿐.)
- 구조: **Zashi 포크**에 메시징 프로토콜을 얹은 형태.
- 키 관리: 지갑에서 두 개의 키 파생.
  - (a) Zcash 거래용 키
  - (b) NOSTR 전용 주소 (이미지/파일 전송용)
- 키 교환: **ECDH → KDF**로 세션 키 도출.
- 더블 래칫(Double Ratchet) 컨셉을 표방하지만 구현이 **잘못됨**.
  - 정상 구현이라면 새 키 파생 후 이전 키를 즉시 삭제해 forward secrecy 확보.
  - 실제로는 root key 파라미터를 그대로 저장해 놓고, n번째 메시지마다 n번 해싱해 키를 재도출 → 사실상 보안 가정이 깨져 있음. 이유는 앱 재설치/디바이스 이동 시 메시지 복구를 위함으로 추정.
- 그룹 채팅은 AES 기반 더 복잡한 키 교환.
- 파일/이미지 전송:
  - 메모 512B 한도를 넘기는 데이터는 청크 분할 + **Blossom / NIP-96 서버**에 AES 암호화 업로드.
  - 메시지에는 접근 path 정보만 포함.
- 타임락/페이먼트(지정 금액 송금 시 메시지 노출) 기능:
  - **스크립트 부재**로 실제 강제력 없음 — UI에서만 가려놓은 수준이라 사실상 무의미.

## 3. Week 4 방향 토론

### 3.1 핵심 아이디어: "프라이버시 크리에이터 플랫폼"

- 컨셉: 크리에이터가 자신의 게시물에 대한 **열람권을 판매**. 일종의 "유료 리딩방 / OnlyFans 류".
- 동기: 익명성 보장이 필요한 창작자(정치적 위협을 피하려는 기자 등)에게 의미 있음. 표현의 자유 / 검열 저항 플랫폼으로서 가치.
- 4가지 핵심 기능 도출:
  1. **키 교환** (E2EE 채널 설정)
  2. **파일 보관** (NOSTR / NIP-96 / Blossom 등 외부 서버 + 암호화 업로드)
  3. **채팅**
  4. **결제** (구독 vs 콘텐츠 단위 원타임)
- 발표 데모 시나리오 아이디어: 발표자가 크리에이터 역할, 청중 일부가 구독자 역할로 라이브 테스트. 기프티콘 사진 등 가벼운 콘텐츠로 시연. 라이브가 어려우면 사전 녹화 영상으로 대체.

### 3.2 결제 모델 비교

| 모델 | 장점 | 단점 |
|---|---|---|
| 원타임(소장형) | 키 로테이션 불필요. 복호키 1회 전달로 구현 단순. | 복호키/계정 공유로 쉽게 우회됨. |
| 구독 | 매월 갱신 강제 → 우회가 어려움. R&D 가치 큼. | 매월 키 교환 필요. 중앙 주체 없이 자동화 어려움. |

→ **원타임을 기본**, 시간 되면 구독을 nice-to-have로.

### 3.3 Zcash로 갈 것인가, 다른 프라이버시 체인으로 갈 것인가

- Zcash의 한계:
  - **스크립트 부재** → bridge/escrow류 신뢰최소화 설계가 사실상 불가.
  - 결제 트랜잭션과 키 교환을 **동기적으로 강제**할 수 없음.
  - 결국 중앙 주체가 키를 들고 중개해야 함.
- 그래도 Zcash를 안고 가고 싶은 이유:
  - **프라이버시의 상징성** — VC들이 Zcash를 사는 이유도 자산성/상징성 측면.
  - Zcash 빌더 생태계가 의도적으로 프로그래머빌리티를 죽여놓았기 때문에 오히려 우리가 시도할 가치.
- 대안 후보:
  - **Aztec** (Noir, zk)
  - **Seismic** (EVM 호환, precompile에 프라이버시 primitive, `suint`로 shielded storage)
  - **Polygon Miden** (EVM 기반이지만 프로그래밍 모델 상이, 아직 테스트넷 추정)
  - **NEAR** 계열의 프라이버시 관련 도구
  - **PAYY** (zk 결제)
- 결론: "GPC 위에서 플랫폼을 만들지" vs "GPC를 떼고 다른 프라이버시 체인으로 갈지"가 핵심 분기점. **양쪽 다 발표 가치는 있음** — Zcash로 시도했지만 막혔다는 서사 자체도 의미 있음.
