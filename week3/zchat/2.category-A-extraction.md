본 섹션은 §1.1 ~ §1.8 walkthrough에서 도출한 evidence를 cross-cut으로 재조합하여, 우리 팀이 **Category A (Zcash Memo Key Exchange Messenger)** 방향으로 갈 때 zchat을 어떻게 reference로 활용할지 정리한다.

## §2.0 Background reading: Memo KEX messenger landscape

### Zcash memo 표준

- **ZIP-302** (Memo Format) — Zcash 노트의 `memo` 필드 정의. 최대 512 bytes UTF-8 또는 raw bytes. ChaCha20-Poly1305로 자동 암호화되어 receiver IVK 보유자만 복호화 가능. memo type byte 컨벤션: 0xF5 (text), 0xF6 (no memo present), 0xF4 (proprietary).
- **ZIP-231** (Memo Bundles, NU7) — 512B 한계를 넘는 메모 묶음을 위한 향후 표준. 단일 노트가 N 개 memo 항목을 가질 수 있게 함. **zchat이 다중 output ZIP-321 multi-output 으로 우회**하고 있는 패턴의 protocol-level 대체. 우리 팀이 NU7 이후 출시한다면 ZIP-321 chunking 코드(§1.5)를 ZIP-231 single-tx-multiple-memos로 단순화 가능.
- **Authenticated Reply Addresses draft** (`draft-ecc-authenticated-reply-addrs`) — ZIP-231 위에 sender authentication signature를 부착하는 향후 표준. zchat이 KEX의 ECDSA 서명으로 우회한 부분의 protocol-level 대체.
- **ZIP-321** (Payment URI) — multi-output transaction을 URI로 표현. `zcash:<addr>?amount=&memo=&address.1=&amount.1=&memo.1=...`. zchat이 메모 청크 송신에 활용(§1.5).
- **ZIP-32** (Hierarchical Deterministic Wallets) — Zcash key derivation. m/32'/<coin>'/<account>' (`coin = 133` for ZEC). 우리 팀이 ZIP-32 spending key 자체에서 ECDH static keypair를 도출하는 차별화도 가능 (§2.4 D-3 참조).


### NOSTR 표준 (file upload 측면)

- **BIP-340 / Schnorr signatures** — NOSTR의 표준 서명 방식. secp256k1 + x-only 32B pubkey. zchat의 NOSTRIdentity가 그대로 사용 (§1.8).
- **NIP-01** — NOSTR event 기본 구조. id, pubkey, created_at, kind, tags, content, sig.
- **NIP-98** (HTTP Auth) — kind 27235. HTTP 요청 인증을 위한 NOSTR event Base64로 `Authorization: Nostr ...` 헤더.
- **NIP-96** (HTTP File Storage Integration) — multipart 파일 업로드 서버 표준.
- **Blossom (BUD-02)** — Blob server 표준. NIP-98 변형의 kind 24242 auth + PUT `/upload`. NIP-96의 단순화 버전.

### Signal Double Ratchet — Reference

zchat의 ratchet 디자인을 평가하는 기준이 되는 baseline:

| 속성 | Signal Double Ratchet | zchat (§1.3) |
|---|---|---|
| Symmetric ratchet (chain key step) | ✓ HMAC-SHA256 | ✓ HMAC-SHA256 (`CHAIN_STEP_BYTE = 0x02`) |
| Message key derivation | `HMAC(chain_key, 0x01)` | `HMAC(chain_key, MESSAGE_KEY_BYTE=0x01)` |
| DH ratchet (재교환) | ✓ — 매 message round-trip마다 새 ephemeral DH | ✗ — KEX 1회로 root 고정 (re-KEX만 root rotation) |
| Forward secrecy (single message) | ✓ | ✓ |
| Post-compromise security | ✓ — DH ratchet으로 회복 | ✗ — root leak 시 영구 |
| Skipped message keys cache | ✓ | ✗ — receiver가 매번 chain을 처음부터 walk (`MAX_SKIP=1000`) |
| Multi-device support | ✓ (linked devices) | ✗ — counter desync |

zchat의 deterministic-root trade-off가 **BIP-39 seed-restore 호환성** 위해 PCS와 skipped-key cache를 포기한 명시적 디자인 선택임을 확인. 우리 팀이 PCS도 원한다면 BIP-39 restore semantic을 포기해야 함 (또는 ratchet state를 별도 backup).

### 한 줄 결론

> zchat은 Section D에서 Memo KEX Messenger 방향의 *유일한 완성된 prior art*다. 메시지 본문을 memo에 담는 모델의 한계(75초 지연 + 영구 onchain + multi-device 부재 + ZEC 비용)가 모두 코드 수준에서 검증된다. 우리 팀이 "memo=KEX only, body=NOSTR transport"로 좁히면 이 한계 4개 중 처음 3개를 회피하며 zchat의 강점(KEX 핸드셰이크, BIP-44 dual derivation, EncryptedSharedPreferences 보안 모델)은 거의 그대로 가져갈 수 있다.

## §2.1 Zchat의 패턴 vs. 우리 팀의 "memo=KEX only / body=NOSTR transport" 가설

### Zchat의 actual model

```
[메시지 본문]
    ↓ E2EMessageProcessor.encryptOutgoing
    ↓ Ratchet (§1.3) → "E2E1:<dir>:<counter>:<ct>"
    ↓ ZMSGProtocol.createV4Reply* (§1.1) → "ZMSG|v4|<convID>|<hash>|E2E1:..."
    ↓ CreateChunkedMessageProposalUseCase (§1.5) → ZIP-321 multi-output
    ↓ Synchronizer (Layer A) → librustzcash → lightwalletd
    ↓
[Zcash transaction memo onchain — receiver IVK로 복호화]
    ↓ SDK transactions Flow
    ↓ ChatViewModel (§1.6) → ZMSG parse + Ratchet decrypt
    ↓
[수신자 UI]

[KEX 핸드셰이크는 같은 transport — ZMSG KEX/KEXACK 메시지로 1회]
[파일 첨부만 별도 NOSTR (Blossom/NIP-96) — URL+hash+wrappedKey 만 memo]
```

본질 한계:
- **75초 평균 블록 지연** (Zcash mainnet)
- **영구 onchain 노출** (receiver IVK leak 시 모든 과거 메시지 노출, 단 Ratchet으로 forward secrecy 일부 회복)
- **메시지당 ZEC 비용** (2000+ zatoshi = 0.00002+ ZEC + ZIP-317 fee — burst send 시 누적)
- **Multi-device 미지원** (counter desync)
- **75초 latency가 typing experience를 깨뜨림**

### 우리 팀의 hypothesis: "memo=KEX only / body=NOSTR transport"

```
[KEX 핸드셰이크 — 1회 per peer]
    Zcash memo path: ZMSG KEX → 75초 지연 OK (한번뿐)
    영구 onchain 노출 = ECDSA-signed pubkey + Bech32 unified address (privacy 측면 안전)
    비용 = ~0.0001 ZEC per peer per relationship

[일반 메시지 — 무한히 자주]
    NOSTR DM (NIP-17 sealed sender 또는 NIP-04 + Ratchet wrapping)
    real-time (~100ms relay propagation)
    off-chain — no permanent record
    free (NOSTR relays 무료 다수)
    multi-device 가능 (sender가 N relays에 publish)
```

### Tradeoffs

| 측면 | Zchat (memo body) | 우리 가설 (KEX only) |
|---|---|---|
| Transport simplicity | 1 채널 (Zcash) | 2 채널 (Zcash + NOSTR) |
| Latency | 75초 + sync (~105초 avg) | ~100ms (NOSTR) |
| Cost per message | ~0.00002+ ZEC | $0 (KEX 1회 비용만 분할) |
| Censorship resistance | ✓ (Zcash 검열 어려움) | NOSTR relay 검열 가능, 다만 multi-relay로 회복 |
| Permanent record | 영구 (Zcash) | 일시 (relay별 retention 정책) |
| Forward secrecy | ✓ Ratchet | ✓ Ratchet (그대로 사용 가능) |
| Post-compromise security | ✗ | DH ratchet 추가하면 가능 (NOSTR identity는 별 keypair) |
| Multi-device | ✗ | ✓ (NOSTR relay 다중 publish) |
| Sender authentication | KEX의 ECDSA + Zcash spend proof | KEX의 ECDSA (Zcash 1회) + NOSTR Schnorr 서명 |
| Privacy from relay operator | Zcash 노트 암호화로 완전 보호 | E2E1 암호화 본문 + NIP-17 sealed sender |

### 검열 저항성 보존 논리

우리 차별화의 핵심 약점은 "NOSTR relay 검열 가능". 그러나 두 가지 이유로 **검열 저항성 핵심은 유지**된다:

1. **KEX의 Zcash 측 보장**: 두 사람이 *서로의 존재를 알게 되는* 핸드셰이크 단계가 censorship-resistant Zcash 위에서 일어남. NOSTR relay가 전부 차단되어도 KEX 자체는 불가능하지 않음.
2. **Multi-relay broadcast**: NOSTR은 정의상 multi-relay. 단일 relay 검열로는 메시지가 막히지 않음. Tor + NIP-17 sealed sender 결합 시 IP-level metadata도 보호.
3. **Fallback path**: NOSTR이 전부 다운된 비상시에는 zchat-style memo body로 fallback 가능 (같은 KEX root + ratchet 사용).

## §2.2 KEX 핸드셰이크 시퀀스 (file:line 인용)

zchat의 KEX 핸드셰이크를 그대로 복제하기 위한 step-by-step reference.

### Alice가 KEX 시작 (KEX initiator)

1. **convId 생성** — `ZchatPreferences.getOrCreateConversationId(peer)` 호출 (atomic at prefs level). 신규면 `ZMSGProtocol.generateConversationId()` (`ZMSGProtocol.kt:139`) — SecureRandom 8자 `[A-Z0-9]`.

2. **E2E keypair 도출** — `E2EEncryption.generateKeyPair()` (`E2EEncryption.kt:137`) — secp256r1 (NIST P-256) `KeyPairGenerator`. Base64 인코딩.

3. **prefs에 keypair 저장** — `zchatPreferences.setE2EOurKeys(peer, pub, priv)` + `setE2EKeyVersion(peer, V2.value)` (V2 = HKDF derivation, `E2EEncryption.kt:24`)

4. **KEX payload 생성** — `E2EEncryption.createKEXPayload(senderAddress, publicKey, privateKey)` (`E2EEncryption.kt:466`)
   - 서명 메시지 = UTF-8(`senderAddress || publicKey_base64`)
   - 서명 알고리즘 = `SHA256withECDSA`
   - 반환 = `"KEX:<pubkey_b64>:<sig_b64>"`

5. **ZMSG envelope** — `ZMSGProtocol.createV4KEXMessage(convId, senderAddress, kexPayload)` (`ZMSGProtocol.kt:190`)
   - 결과 = `"ZMSG|v4|<convID>|KEX|<senderHash16>|KEX:<pubkey>:<sig>"`
   - senderHash = SHA-256(senderAddress)[0..8] hex (`ZMSGProtocol.kt:106`)

6. **Zcash 송신** — `CreateChunkedMessageProposalUseCase.invoke(rawMemo = true, destinationAddress = peer)` (`CreateChunkedMessageProposalUseCase.kt:64`). amountPerOutput = 1000 zatoshi default.

7. **ChatViewModel이 자기 송신 KEX 의 txid 캐싱** (`ChatViewModel.kt:668-682`) — `zchatPreferences.setE2EKexTxId(peer, txIdStr)` — 나중 ratchet root 도출에 필요.

### Bob 수신 + 검증 + KEXACK 송신

8. **수신 — SDK Synchronizer Flow** — `transactions.filterNotNull().debounce(300)` (`ChatViewModel.kt:493`) 가 새 tx 발견 → `convertToConversations(transactions, ...)` (`ChatViewModel.kt:570`)

9. **분기 — KEX 메시지** — `ZMSGProtocol.isKEXMessage(memoText)` true (`ChatViewModel.kt:663`) → `handleKEXMessage(memoText, userAddress, txIdStr)` (`ChatViewModel.kt:1702`)

10. **KEX parse + verify** — `ZMSGProtocol.parseKEXMessage(memo) → (convId, kexPayload)` (`ZMSGProtocol.kt:239`) → `E2EEncryption.parseKEXPayload(kexPayload, senderAddress)` (`E2EEncryption.kt:480`)
    - parse: prefix `"KEX:"` 검사 + colon split 2 parts
    - verify: `Signature.getInstance("SHA256withECDSA").verify(...)` (`E2EEncryption.kt:436`)
    - 검증 실패 시 null 반환 → handle 종료

11. **peer pub key 저장** — `zchatPreferences.setE2EPeerPublicKey(senderAddress, peerPublicKey)` (`ChatViewModel.kt:1820`)

12. **자동 E2E enable** (`ChatViewModel.kt:1822-1830`):
    ```kotlin
    if (!isE2EEnabled(senderAddress)) {
        if (getE2EOurPublicKey(senderAddress) == null) {
            val keyPair = E2EEncryption.generateKeyPair()
            setE2EOurKeys(senderAddress, keyPair.publicKey, keyPair.privateKey)
            setE2EKeyVersion(senderAddress, E2EKeyVersion.V2.value)
        }
        setE2EEnabled(senderAddress, true)
    }
    ```

13. **KEXACK 송신** — `sendKEXAckMessage(peerAddress, ourAddress, convId)` (`ChatViewModel.kt:1899`)
    - `createKEXAckPayload` (`E2EEncryption.kt:509`) — `"KEXACK:<pub>:<sig>"`
    - `createV4KEXAckMessage(convId, ourAddress, kexAckPayload)` (`ZMSGProtocol.kt:203`)
    - `createChunkedMessageProposal(rawMemo = true)` 호출

14. **자기 송신 KEXACK 의 txid 캐싱** (`ChatViewModel.kt:679`) — `setE2EKexAckTxId(peer, txIdStr)`

### 양쪽 — Ratchet root 도출

15. **ECDH shared secret 계산** — `E2EEncryption.deriveSharedSecret(ourPriv, peerPub, V2, psk = null)` (`E2EEncryption.kt:159`) **단, raw shared secret 만 추출** (Ratchet root에 별도 derivation 적용하므로). 실제 호출 패턴은 ChatViewModel `getOrCreateMessageProcessor` 안에서 결정 — 일부 코드 reading 필요.

16. **Ratchet root 도출** — `E2ERatchet.deriveRatchetRoot(ecdhSharedSecret, psk, kexTxidBytes, kexAckTxidBytes)` (`E2ERatchet.kt:247`)
    ```kotlin
    val ikm = if (psk != null) ecdhSharedSecret + psk else ecdhSharedSecret
    val kexContext = sha256(kexTxid + kexAckTxid)
    return HKDF.deriveKey(
        ikm = ikm,
        salt = "ZCHAT_RATCHET_ROOT_V1".toByteArray(),
        info = kexContext,
        length = 32,
    )
    ```

17. **Direction byte 결정** — `isLower = (ourPub_compressed < peerPub_compressed)` (lex compare of compressed secp256r1). caller(ChatViewModel)이 계산. *0x00 chain* = lower→higher direction, *0x01* = higher→lower.

18. **E2EMessageProcessor 인스턴스 캐시** — `E2EMessageProcessor(rootKey, convId, isLower, EncryptedPrefsRatchetStateStore)` (`E2EMessageProcessor.kt:18`) — peer별 캐시 (`ChatViewModel.getOrCreateMessageProcessor`)

### 이후 메시지 (다 ratchet)

19. **송신**: `processor.encryptOutgoing(plaintext)` (`E2EMessageProcessor.kt:30`) → `"E2E1:<dir>:<counter>:<ct>"` → ZMSG envelope → ZIP-321 → Zcash

20. **수신**: ZMSG parse → `processor.decryptIncoming("E2E1:...")` (`E2EMessageProcessor.kt:44`) → plaintext

## §2.4 차별화 방향 (D-1 ~ D-5)

각 differentiation 후보는 **(설명) → (zchat code reference) → (week2 ★ project evidence)** 형식.

### D-1: Memo = KEX only / Body = NOSTR DM (NIP-17 Sealed Sender)

**아이디어:** 본 §2.1의 핵심 가설을 그대로 구현. KEX 1회만 Zcash memo. 메시지 본문은 NOSTR DM으로.

**zchat reference:**
- KEX 송신/수신 흐름: `ChatViewModel.kt:1849, 1899, 1702`
- NOSTR identity 도출: `NOSTRIdentity.kt:127`
- Ratchet wire format (NOSTR DM에 그대로 적용 가능): `CiphertextWireFormat.kt:21`

**Week2 evidence:**
- ★★ `Zeke` (#69) — NOSTR이 메시지 channel로 가능함을 확인하는 PoC (단방향 서비스 명령이지만)
- ★★ `Mumtaz / zec2eth` (#61) — viewing key 기반 memo decode 모델 (KEX 검증에 응용 가능)
- ★ `Raven Bridge` (#56) — memo에 routing identifier만 박는 minimal pattern

**Trade-off:** 2-channel 복잡도 ↑, latency ↓ (~100x faster), cost → 0, censorship 약화 (NOSTR relay 검열 가능)

### D-2: ZIP-321 multi-recipient KEX broadcast (그룹 first contact 효율화)

**아이디어:** 그룹 멤버 N명에게 *한 번의 Zcash transaction*으로 KEX 발사. 각 멤버용 ECIES wrap 1개씩 + same convID. 멤버는 동시에 받아 같은 timestamp로 그룹 합류.

**zchat reference:**
- ZIP-321 multi-output: `CreateChunkedMessageProposalUseCase.kt:213-249`
- ECIES per-member wrap: `E2EEncryption.kt:569, 749`
- Group fan-out (현재 N개 별도 tx): `GroupViewModel.kt:551-607`

**Week2 evidence:** Section A `t2z` 같은 multi-output 패턴 + zchat의 group invite 코드 결합

**Trade-off:** transaction 비용 N배 → 1배 감소. atomicity 강화. memo 512B 한계로 N ≤ ~4 (멤버당 ECIES 90 bytes 가정)

### D-3: Viewing-key 기반 KEX receipt (selective disclosure)

**아이디어:** KEX가 끝난 후 *KEX 자체에 대한 receipt*을 생성. KEX/KEXACK txid + viewing key proof를 hash해서 "이 두 사람이 X 시점에 KEX 했음"을 third party에 selectively disclose 가능.

**zchat reference:**
- KEX/KEXACK txid binding to ratchet root: `E2ERatchet.kt:247-261`
- ZIP-32 derivation from BIP-39 seed (viewing key 도출): Layer B `ZashiSpendingKeyDataSource`

**Week2 evidence:**
- ★★★ `PCZTKit` (#62) — PCZT (ZIP-374) multi-party signing — viewing key 기반 disclosure에 직접 reference
- ★★★ `Miden Zcash` (#57) — ivk + Sapling note decrypt — TypeScript viewing key decode 직접 코드
- ★★★ `Zscreener` (#66) — viewing key를 Nillion Secret Vault에 저장 — 보관 모델

**Trade-off:** compliance / audit / proof-of-conversation 같은 use case가 추가 가능. 일반 채팅 user에게는 unused feature.

### D-4: Identity Regeneration + 자동 ADDR migration

**아이디어:** zchat이 `IdentityManager` + `createV4ADDRMessage`를 가지고 있으나 *명시적 ADDR broadcast*는 자동화되어 있지 않음. 우리는 새 identity 만들 때 기존 contacts 전체에 ADDR 메시지 자동 발사 + receipt 추적.

**zchat reference:**
- IdentityManager (DEC-016): `IdentityManager.kt:200-235`
- `createV4ADDRMessage` wire format: `ZMSGProtocol.kt:305`
- `parseADDRMessage`: `ZMSGProtocol.kt:334`

**Week2 evidence:** Section A `Zucchini Wallet` (#15) — UI/UX + key 파생 reference

**Trade-off:** UX 매끈 ↑, 추가 transaction 비용 (contacts N개 × 0.00001 ZEC). 자동 broadcast가 사용자 의도 없이 발사되는 위험 → 명시적 confirm 필요.

### D-5: Hardware-backed E2E keys (Android Keystore key-agreement API)

**아이디어:** zchat의 *명시적 미해결 gap* (ZMSG_PROTOCOL_SPEC.md Known Gaps). E2E keypair가 현재 EncryptedSharedPreferences (Tink로 보호) 에 저장되지만 *Android Keystore key-agreement API*를 쓰면 priv key가 디바이스 메모리에 절대 노출되지 않음. ECDH도 hardware-attested.

**zchat reference:**
- 현재 E2E keypair 저장: `ZchatPreferences.setE2EOurKeys`
- HKDF/AES-GCM helpers: `E2EEncryption.kt`

**Week2 evidence:** 명시적으로 hardware-backed key를 사용한 Section D 프로젝트 없음 — **우리 팀이 first mover 가능**.

**Trade-off:** Android Keystore API 학습 + StrongBox 지원 OS 한계 (Android 9+, 일부 디바이스만 StrongBox HSM). seed restore semantic이 더 복잡해짐 (Keystore 키는 device-bound — backup 시 별도 export 흐름 필요).

## §2.5 한 줄 결론

> zchat에서 그대로 가져갈 자산 = (a) KEX 핸드셰이크 wire format + ECDSA 검증, (b) Symmetric ratchet (`E2ERatchet`) + state store, (c) NOSTRIdentity BIP-44 derivation, (d) Quantum Shield PSK exchange, (e) EncryptedSharedPreferences + PBKDF2 + collision-guarded AddressCache. **이 다섯 자산은 NOSTR transport 위에서도 그대로 작동한다.** 다시 만들어야 할 것 = ZMSG 메시지 본문 송신 path 전체와 60초 폴링 + 옵티미스틱 UI를 NOSTR relay subscription + real-time receipt로 교체. 그 결과는 "검열 저항성은 유지하면서 75초·영구·multi-device 한계를 회피한 zchat-evolved 메신저"다.
