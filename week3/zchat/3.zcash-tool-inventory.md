### lightwalletd 직결의 의미

zchat은 **자체 백엔드(api.zsend.xyz)를 wallet path에 호출하지 않는다** (§1.6 Q15 검증). 즉:
- Synchronizer가 직접 lightwalletd gRPC endpoint에 연결 (`sdk-ext-lib/.../LightWalletEndpoint`에서 정의된 default endpoint)
- 사용자가 endpoint를 변경하면 own lightwalletd 또는 다른 community-run server 가능
- Tor / VPN 사용 시 IP-level metadata 추가 보호 (zchat 자체엔 Tor integration 없음 — ZMSG_PROTOCOL_SPEC Known Gaps에 명시)

## §3.2 ZIP-32 derivation 경로 (BIP-44 dual derivation 검증)

Week2 메모의 핵심 claim: "m/44'/133' Zcash UA + m/44'/1237' NOSTR secp256k1 동시 파생".

### NOSTR 측 (검증됨)

§1.8에서 검증: `NOSTRIdentity.kt:115-122`
```kotlin
private val DERIVATION_PATH = intArrayOf(
    44 or HARDENED_FLAG,     // purpose
    1237 or HARDENED_FLAG,   // coin type for NOSTR
    0 or HARDENED_FLAG,      // account
    0,                       // change
    0                        // index
)
// = m/44'/1237'/0'/0/0
```

- BIP-32 HMAC-SHA512 ("Bitcoin seed", seed) → master
- 각 path segment에서 hardened/normal 분기 처리
- 최종 priv key + Schnorr public key (BIP-340 x-only 32B)
- Bech32 encode → `npub1...`

### Zcash 측 (Layer B 검증 필요 — 본 dive scope outside)

Zcash UA는 BIP-44가 *아니라* **ZIP-32**를 사용:
- Path = `m/32'/<coin>'/<account>'`
- coin = 133 (Zcash mainnet) 또는 1 (testnet)
- account = 0 (zchat은 single account)
- USK = HKDF over ZIP-32 의 raw key → Orchard + Sapling + Transparent 모두 도출

> **week2 메모의 "m/44'/133'" 표현은 정확히 말하면 BIP-44가 아니라 ZIP-32.** BIP-44는 secp256k1 + Bitcoin 컨벤션이고 Zcash UA는 Pallas/Jubjub 등 다른 curve를 쓰므로 BIP-44 표준에 fit하지 않음. 그러나 *coin type 133*은 BIP-44에서 Zcash에 할당된 슬롯과 *우연히* 일치 (ZIP-32가 BIP-44 슬롯을 borrow). 정확한 wording: **"같은 BIP-39 seed에서 BIP-44 path (NOSTR) + ZIP-32 path (Zcash) 양쪽 모두 도출."**

### 같은 seed 공유의 의미

- Backup 한 BIP-39 mnemonic 24 단어만 있으면 두 identity (Zcash UA + NOSTR npub) 모두 복원 가능
- 디바이스에서 두 derivation은 *독립적* — 한쪽 curve가 깨져도 다른 쪽은 안전
- 우리 팀이 NOSTR DM으로 메시지 본문을 보내는 가설(§2.1) 채택 시 *추가 mnemonic 없이* 두 identity 활용 가능

## §3.3 Orchard-only 정책

`MESSAGING_CRYPTO.md` §4.1 명시: "Orchard 잔액 확인 (메시지는 Orchard pool 만 사용)".

### 코드 검증

- `ChatViewModel.sendMessage` (§1.6 line 2312)이 Orchard 잔액 확인 후 진행 (정확한 line은 본 dive 부분 read scope 외 — `accountDataSource.spendableShieldedBalance` 같은 API 사용)
- `CreateChunkedMessageProposalUseCase` 자체는 pool 선택을 강제하지 *않음* — SDK가 노트 선택 (§1.5 Q13)
- 즉 *application-level 정책*: ChatViewModel 진입 시 Orchard 잔액 검증, Sapling-only 사용자는 메시지 송신 불가 (또는 shielding 안내)

### Orchard-only를 채택한 이유 (추정 + spec)

1. **Halo2 zk-SNARK**: Groth16 (Sapling) 대비 trusted setup 불필요, smaller proof size, faster verify
2. **ZIP-317 fee policy**: Orchard transparent 비율 변경 안 함 → fee 예측 가능
3. **anonymity set 증가**: 시간이 갈수록 Orchard pool size 가 커짐 (Sapling은 ZSA 도입 후 deprecated 예상)
4. **memo 사이즈는 양쪽 동일**: 512B 한계는 Sapling/Orchard 공통이므로 ZMSG 측에서는 indifferent

### 우리 팀에 미치는 영향

Category A 구현 시 Orchard-only 전제 그대로 채택 권장. 사용자가 Sapling 잔액만 있으면:
- "messaging 전 fully shielded into Orchard" UX 흐름 (zchat이 PrivacyStatus.needsShielding으로 표시 — `ChatMessage.kt:54`)
- 또는 Sapling → Orchard pool transfer 자동화

## §3.4 ZIP-321 multi-output 사용 깊이

§1.5에서 자세히 검증.

핵심 사실:
- **모든 메시지가 ZIP-321 multi-output 사용** — chunk 개수 1이어도 platform fee output 1개가 추가되므로 항상 ≥ 2 outputs
- URI 형식: `zcash:<addr>?amount=...&memo=...&address.1=<addr>&amount.1=...&memo.1=...` (indexed parameter)
- memo encoding: `Base64.URL_SAFE | NO_WRAP | NO_PADDING`
- 모든 메시지 output은 *같은 destination address*로 송신 (chunk마다 다른 ephemeral 주소 사용 안 함)
- platform fee output: `ZMSGConstants.PLATFORM_FEE_ADDRESS` 하드코딩된 unified address

ZIP-321의 다른 활용 가능성 (zchat이 사용하지 않는):
- multi-payee transactions (한 tx로 여러 명에게 송금) — 그룹 fan-out에 적용 가능했지만 zchat은 N개 별도 tx 선택 (§1.4 노트)
- `req-` prefix label (zchat 미사용)
- Sapling/Orchard pool selection hint (zchat 미사용)

## §3.5 Synchronizer 호출 패턴 — 3-step proposal pattern

zchat은 SDK의 *standard 3-step proposal*을 따른다:

```
1. proposeSend / proposeTransferFromUri  → Proposal (서명 X)
2. createProposedTransactions(proposal, usk) → ByteArray (signed serialized tx)
3. lightwalletd.submitTransaction         → tx hash
```

이 3단계는 SDK 내부에 묶여 있으며 zchat은 Layer B repository(`ZashiProposalRepository.submit()`)로 한 번에 호출. proposal과 submit 사이에 user review screen이 들어갈 수 있는 흐름 (review path)도 SDK가 지원. zchat의 ZCHAT 메시지는 `directSubmit = true`로 review 우회.

PCZT 분기 (Keystone 하드웨어 지갑):
- proposal 생성 후 PCZT 추출 (`createPCZTFromProposal`)
- 사용자가 Keystone 디바이스로 PCZT 서명
- 서명된 PCZT를 다시 받아서 submit

본 dive scope (Zashi 계정만 다룸)에서는 PCZT 흐름 detail 외.

## §3.6 EncryptedSharedPreferences (Android Keystore) 키 저장 모델

### 두 layer 의 별도 storage

```
[Layer B 백업 영역]
co.electriccoin.zcash.encrypted  ← mnemonic + USK input (key namespace)
   ↓ EncryptedSharedPreferences (Tink-backed)
   ↓ MasterKey AES256_GCM, PrefKeyEncryption AES256_SIV, PrefValue AES256_GCM
   ↓ MasterKey backed by AndroidKeyStore (TEE 또는 StrongBox HSM)

[Layer C zchat 영역 — 별도 prefs files]
zchat_address_cache         ← hash→address 매핑 (AddressCacheImpl)
zchat_contact_book          ← contact list (ContactBookImpl)
zchat_identity_manager      ← Identity (mask) list (IdentityManagerImpl)
zchat_preferences           ← E2E keypair, ratchet state, group keys, PSK,
                              convID mappings, drafts, status, mute, settings
                              (ZchatPreferences — 1801 lines)
                              ★ 일부 항목만 EncryptedSharedPreferences (E2E priv key
                              포함). 정확한 분리는 코드 추가 확인 필요
```

### Tink + Android Keystore 보호 메커니즘

1. **MasterKey** = Android Keystore에 저장된 AES-256 키. TEE (Trusted Execution Environment) 또는 StrongBox HSM에 의해 보호. **앱 코드도 key bytes에 접근 불가** — 사용 시 Keystore에 ciphertext를 보내 복호화/암호화만 요청.
2. **Pref key encryption** = AES256_SIV (deterministic — 같은 키는 항상 같은 ciphertext, search 가능)
3. **Pref value encryption** = AES256_GCM (12B random nonce — value는 매번 다른 ciphertext)
4. **결과**: 디바이스 rooting/imaging 없이는 prefs 데이터 추출 불가. 디바이스 도난 + screen lock bypass 시도 시 StrongBox 가 attestation 검증.

### zchat이 안 쓰는 / 약점

- **E2E priv key를 Android Keystore *key-agreement API*로 직접 보호하지 않음** — Tink wrapped로 EncryptedSharedPreferences에 raw bytes 저장. Keystore-managed EC key는 export 불가능한 priv key + hardware-attested ECDH 가능 (Android 9+) — zchat 미채택, ZMSG_PROTOCOL_SPEC.md Known Gaps에 명시.
- **PIN/Phrase 보호는 PBKDF2 600k (SecureHash, §1.8)** — Keystore PIN flow 가 아닌 application-level. 강도는 OWASP 2023 권장치 충족이지만 hardware-bound 아님.

## §3.7 우리 팀이 추가로 도입 검토할 도구

zchat이 *사용하지 않지만* Category A 진짜 구현 시 검토할 도구들. 각각 one-liner + week2 ★ project reference.

### A. PCZT (`pczt` crate, ZIP-374)

- **What**: Partially Constructed Zcash Transaction. Multi-party signing role separation (Creator/Constructor/Prover/Signer/Combiner/Finalizer).
- **Why for us**: KEX 자체는 1인 서명이지만 *그룹 KEX broadcast* (§2.4 D-2)에서 multi-party signing이 필요한 경우. 또한 hardware wallet 통합 (§2.4 D-5 변형)에도 직접 활용.
- **Week2 reference**: ★★★ `PCZTKit` (#62), Section A의 t2z 2개도 같은 `pczt` crate 사용.

### B. lightwalletd direct (Tonic gRPC, Rust 백엔드)

- **What**: zcash-android-wallet-sdk를 우회하고 Rust 백엔드가 lightwalletd에 직접 연결.
- **Why for us**: Android 앱 외에 server-side bot (예: bot peer 가 메시지 보내거나 KEX 자동 acknowledge)을 만들 때. SDK는 mobile 전용이므로 server에서 못 씀.
- **Week2 reference**: ★★ `Mumtaz / zec2eth` (#61) — Rust watcher가 `zcash_client_backend` lightwalletd-tonic 직접 사용. ★ `Zeke` (#69) — Zingo + lightwalletd proto.

### C. Viewing key 기반 receipt + selective disclosure

- **What**: Outgoing viewing key (ovk) 또는 Incoming viewing key (ivk)를 third-party와 공유하여 "내가 이 transaction을 받았다/보냈다"를 selectively prove.
- **Why for us**: §2.4 D-3 differentiation. 또한 compliance/audit 시나리오 — 사용자가 자기 메시지 history를 *증명*하지만 본문은 숨기는 use case.
- **Week2 reference**: ★★★ `Zscreener` (#66) — viewing key를 Nillion Secret Vault에 저장. ★★★ `Miden Zcash` (#57) — ivk + Sapling note decrypt TypeScript 직접 구현. ★ `Zcash Analystics` (#65) — 클라이언트에서만 viewing key 보관.

### D. ZIP-231 memo bundles (NU7 — Network Upgrade 7)

- **What**: 512B 메모 한계를 넘는 multi-memo bundle. 단일 노트가 multi memo를 가질 수 있음.
- **Why for us**: zchat의 ZIP-321 chunking (§1.5) 대체. *KEX 자체*에 ECDSA 서명 + ECDH pubkey + chain code + 기타 metadata를 모두 한 memo에 넣을 수 있음 → 512B 한계 회피.
- **Status**: NU7 활성 시점 (2026 또는 2027 예상) 이후. 우리 팀이 NU7 이후 출시할지에 따라 결정.
- **Week2 reference**: 없음 — zchat이 첫 mover 가능.

### E. ML-KEM-768 hybrid (Real PQ KEX)

- **What**: NIST FIPS 203 (CRYSTALS-Kyber). 실제 post-quantum key encapsulation mechanism. ECDH(secp256r1) || ML-KEM-768 hybrid 가 Signal PQXDH 모델.
- **Why for us**: zchat이 명시적으로 deferred한 "Real PQ milestone". Quantum Shield는 PSK augmentation일 뿐 (§1.2 노트).
- **Constraint**: ML-KEM-768 public key = 1184B, ciphertext = 1088B. 모두 512B memo 한계 초과 → ZIP-231 또는 chunked KEX 필요. 우리 NOSTR transport 가설에선 NOSTR DM 본문에 들어가므로 size 제약 없음. **우리 팀이 NOSTR transport 채택 시 ML-KEM-768 hybrid가 더 자연스러움.**
- **Week2 reference**: 없음 — 마찬가지 first-mover 기회.

### F. Tor / SOCKS5 transport for lightwalletd

- **What**: lightwalletd gRPC connection을 Tor proxy로 routing.
- **Why for us**: IP-level metadata 보호. zchat ZMSG_PROTOCOL_SPEC.md Known Gaps에 "Native IP-level metadata protection 부재".
- **Week2 reference**: 없음 — implementation cost vs benefit trade-off 검토.

### G. NIP-17 Sealed Sender (NOSTR private DM)

- **What**: NOSTR DM의 sender authentication을 hide. NIP-04의 약점 (sender pubkey 공개) 해결.
- **Why for us**: §2.4 D-1 핵심. NOSTR 채택 시 사용자 metadata 보호의 핵심.
- **Week2 reference**: 없음 — NOSTR side이므로 Zcash project list 외부.

## §3.8 한 줄 결론

> zchat은 **mobile boundary 안에서** Zcash native 통합이 완성된 첫 번째 messenger다. 그러나 *hardware-backed key 보호*, *PCZT multi-party signing*, *viewing-key selective disclosure*, *ZIP-231 NU7 memo bundles*, *Real PQ (ML-KEM hybrid)*, *Tor*, *NIP-17 sealed sender* 등은 모두 미채택 자산이다. 우리 팀이 Category A로 가면 이 7개 중 최소 3개 (특히 D-1 NIP-17 + D-5 Keystore 키 + D-3 viewing-key receipt) 를 채택해 zchat에서 한 단계 진화한 메신저를 만들 수 있다.
